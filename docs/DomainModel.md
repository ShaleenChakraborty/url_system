# 🗂️ Domain Model

> _"The Domain Model represents the core business objects of the application and how they evolve throughout a request lifecycle."_

---

# Overview

The current version of **URL System** revolves around a single domain entity:

```
Url
```

Although simple, this entity represents the central business object responsible for mapping an original URL to a shortened identifier while tracking usage statistics.

Future versions will expand the domain with additional entities such as analytics events, users, API keys, and rate limiting metadata.

# Domain Lifecycle

Every URL follows the same lifecycle.

```text
Client

↓

Submit Original URL

↓

Generate Short Code

↓

Persist in PostgreSQL

↓

Serve Redirect Requests

↓

Increment Click Count

↓

Synchronize Redis Cache

↓

Serve Analytics
```

---

# URL Entity

```
@Table(name = "urls")
```

The `Url` entity is the application's primary business object.

Each record represents a permanent mapping between an original URL and its generated short code.

# Field Breakdown

## id

```java
Long id
```

Purpose

- Primary Key
- Generated automatically
- Internal database identifier

Example

```
15
```

---

## originalUrl

```java
String originalUrl
```

Stores the original destination URL.

Example

```
https://openai.com
```

This is the final URL users are redirected to.

---

## shortCode

```java
String shortCode
```

Unique Base62 identifier generated by the application.

Example

```
Ab9X2K
```

This becomes

```
http://localhost:8080/Ab9X2K
```

---

## clickCount

```java
Long clickCount
```

Tracks the number of successful redirects.

Current update strategy

```text
Redirect

↓

clickCount++

↓

Save PostgreSQL

↓

Refresh Redis
```

# Entity Lifecycle

## Creation

```text
POST /shorten

↓

Generate Base62 Code

↓

Create Url Entity

↓

Persist
```

---

## Retrieval

```text
GET /{shortCode}

↓

Redis

↓

Database (if required)

↓

Return Url
```

---

## Update

The entity is updated after every redirect.

```text
Retrieve URL

↓

Increment Click Count

↓

Persist

↓

Refresh Redis
```

---

# Data Ownership

```text
PostgreSQL
      │
      ▼
Source of Truth
```

Redis never owns application data.

Redis stores temporary cached copies.

---

# DTO Layer

The application intentionally separates persistence models from API models.

```text
Client

↓

Request DTO

↓

Controller

↓

Service

↓

Entity

↓

Repository

↓

Database
```

Response flow

```text
Database

↓

Entity

↓

Service

↓

Response DTO

↓

Client
```

This prevents exposing internal database models directly to API consumers.

---

# Current DTOs

## UrlRequest

Purpose

Receives client input.

```text
Client

↓

POST /shorten

↓

UrlRequest
```

Validation

- URL cannot be empty
- Must begin with http:// or https://

---

## UrlResponse

Purpose

Returns the shortened URL.

Contains

- originalUrl
- shortUrl

---

## AnalyticsResponse

Purpose

Returns analytics information.

Contains

- Original URL
- Short Code
- Short URL
- Click Count

---

## ErrorResponse

Standard error payload returned by the Global Exception Handler.

Contains

- Timestamp
- HTTP Status
- Error
- Message
- Request Path

# Summary

The current domain model is intentionally simple.

The `Url` entity acts as the central business object while DTOs isolate the REST API from persistence concerns.

As the application evolves, new domain objects will be introduced around this core without breaking existing responsibilities.
